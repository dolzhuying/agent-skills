# Skill 工程质量指南

本指南从 8 个生产级 Skill 的实战经验中提炼而来，覆盖 Skill 的**创建**和**审查**全流程。核心哲学：

> **用确定性规则约束 AI 的不确定性，用显式失败替代静默错误，用工具验证替代 AI 推断，用二次确认保护不可逆操作。**

## 使用方式

本指南有两种使用模式：

**创建模式**：在 skill-creator 流程中编写 SKILL.md 初版时，参考下方的骨架模板和各章节的工程规范，边写边对照，确保初版就具备生产级结构。

**审查模式**：SKILL.md 写完或迭代后，用末尾的检查清单逐项验证，查漏补缺。

不是每条规则都适用于每个 Skill——简单工具型 Skill 只需关注结构和触发条件，复杂业务型 Skill 则需要全面审查。

---

## 零、骨架模板（创建时直接套用）

创建新 Skill 时，以此骨架为起点填充内容。章节顺序经过验证，不可随意调整：

```markdown
---
name: <skill-name>
description: |
  一句话定位（是什么，做什么）。

  触发条件（满足任一）：
  1. [意图动词 + 括号内原话举例]
  2. [意图动词 + 括号内原话举例]

  不触发（留给 <other-skill>）：
  - [排除场景1]
  - [排除场景2]
---

# <Skill 名称>

## 核心规则（最高优先级）
- **禁止**：[具体行为] | 原因：[为什么] | 替代：[该怎么做]
- **禁止**：...

## 依赖清单

| 类型 | 名称 | 用途 | 必须/可选 |
|------|------|------|-----------|
| 脚本 | scripts/xxx.py | ... | 必须 |
| Skill | other-skill | ... | 可选 |

## 场景与执行 SOP

### 场景一：[场景名]
**触发**：[什么情况进入此场景]
**步骤**：
1. ...
2. ...

### 场景二：[场景名]
...

## 命令速查

| 需求 | 命令 |
|------|------|
| 做A | `python3 scripts/xxx.py --param value` |

## 参考文档（按需读取）

| 场景 | 文件 |
|------|------|
| [复杂场景A] | `references/A.md` |
```

以下各章节解释骨架中每个部分的工程规范和背后的原因——创建时参考，审查时对照。

## 一、文档结构

### 标准章节顺序

SKILL.md 应按以下顺序组织章节，不可随意调整：

```
1. Front Matter YAML（name + description，含触发/不触发条件）
2. 核心规则 / 刚性约束
3. 依赖 Skill / 脚本 / 工具清单（表格化）
4. 场景与执行 SOP
5. 命令速查 / 快速开始
6. 参考文档索引
```

为什么这个顺序：AI 必须先内化行为边界（规则），再看可用工具（清单），再理解怎么用（SOP）。如果规则放在最后，AI 会先"想用什么"再回头看"能不能用"，规则遵从率会下降。

### 分层架构：按需加载

SKILL.md 只放 AI 每次都需要的信息。低频、高复杂度的内容拆到 `references/` 目录，用条件引用：

```markdown
当遇到大看板场景时，读 references/execution-rules.md 获取完整细则。
```

拆分阈值：单场景 SOP >30 行、字段映射表 >20 行、错误模式 >10 条。SKILL.md 总体 150-400 行为宜。

为什么这么做：LLM 注意力是有限资源。信息越多，关键规则的遵从率越低。按需加载让 AI 在简单路径上保持精准。

## 二、触发条件

### 正向触发：意图动词 + 原话举例

用意图动词保证语义覆盖面，用括号内原话提供锚点：

```yaml
# 好
用户想对比渠道/代理/账户的投放效果（如"哪个渠道性价比好""东信和米瑟比怎么样"）

# 差——覆盖率低，换种说法就失效
用户说「厂商日报」「厂商渠道日报」
```

为什么：意图动词让 LLM 做语义匹配而非字面匹配，原话示例提供校准样本。纯关键词白名单的召回率不够。

### 负向排除：必须写，且注明"留给谁"

```yaml
不触发（留给 vendor-eco-analyst）：
- 用户明确说"生态"或"厂商生态"
```

为什么：没有负向排除的 Skill 在多 Skill 环境下必然冲突。标注"留给谁"让 AI 在排除后能给用户明确引导，而非迷茫。

### 模糊场景：内嵌默认路由

```yaml
用户只提到厂商名但未提及"生态" → 默认按付费投放查询
```

为什么：路由歧义留到运行时，AI 要么猜错要么反问，都影响体验。在定义层给出默认值消除不确定性。

### 管道中游：用数据状态触发

如果 Skill 处于协作链中游，触发条件应基于上下文中是否存在合法数据对象，而非用户意图：

```yaml
仅当对话中已有 dataset_list 且用户要求正式取数时触发
```

为什么：管道中游被自然语言直接触发会跳过上游必要步骤，执行基础不牢。数据状态作为门控相当于类型检查。

---

## 三、AI 与脚本的职责分离

### 核心原则：AI 编排，脚本计算

AI 负责：意图识别、参数提取、调用顺序编排、成功信号检查、用户沟通。
脚本负责：所有数值计算、HTML 渲染、API 调用、数据聚合。

为什么：AI 的生成能力是概率性的。数据计算等要求 100% 精确的场景，委托给确定性脚本后，输出质量不再受 prompt 变化或模型版本影响。

### 确定性渲染分离

```
AI 准备数据 (JSON) → 渲染脚本 → 固定输出
```

为什么：AI 直接生成 HTML/报告时每次格式可能有差异。用渲染脚本后，只要 JSON 正确，输出就是确定的。样式迭代只改脚本，不动 prompt。

### 命令可直接复制执行

SKILL.md 中的脚本调用必须是完整命令，不能是伪代码：

```bash
python3 scripts/build_report.py --date 20260419 --output output/report.json
```

为什么：AI 在"理解描述→生成命令"时有出错风险。直接给出可执行命令后，AI 只需做参数值替换，出错空间大幅缩小。

## 四、安全与幻觉防护

### 禁止条款三层结构

每条禁令都包含：**禁止什么 + 为什么禁止 + 替代方案**。

```
禁止：预估 SQL 执行时间
原因：进度与耗时完全非线性，预估会误导用户
替代：只转述事实——"当前进度 45.0%，已耗时 120 秒"
```

为什么：LLM 理解了原因后，即使遇到 prompt 未覆盖的变体场景也能正确判断。没有原因的禁令在边缘场景下容易被违反。

### 把已知缺陷写进 prompt

直接用历史 bug 案例证明 AI 的弱点，比泛泛说"要准确"有效得多：

```
你对日期和星期的心算一定是错的。
真实案例：time_context 返回「上周: 04-20 至 04-26」，
agent 自己写了「04-21 ~ 04-27」（那是"近7天"不是"上周"）。
```

为什么：AI"认为"自己算得准，泛泛规则无约束力。用真实案例打破这种自信后，它会倾向于使用你提供的替代工具。**承认缺陷 + 提供工具 > 要求准确**。

### 显式失败优于静默降级

```
# 差：图表没有该字段 → 悄悄换 NL 取数
# 好：明确提示"当前图表不包含该维度，建议换用支持该维度的图表"
```

为什么：静默降级最危险——用户以为请求正确处理了，实际数据口径完全错误。显式失败的成本是一次交互中断，静默降级的成本可能是错误的业务决策。

### 领域知识防护

如果 Skill 涉及有时序特性的数据，内嵌延迟/回收规则防止 AI 把"数据未到"误判为"数据异常"。这类领域知识 AI 不具备，必须显式注入。

## 五、工作流编排

### 成功信号约定

每个关键步骤必须有固定字符串标志，AI 用字面匹配判断成功：

```
build_report.py → stdout 包含 "✅ 报告生成完成"
```

为什么：脚本 stdout 混合了日志、警告等内容。靠 AI 语义理解判断是否成功不可靠，固定标志让判断变成字面匹配。

### 产物命名约定

文件名必须可预测，使 AI 无需解析 stdout 就能拼出下一步的输入路径：

```
{动作}_{维度}_{日期}.{格式}
data_全渠道_20260419.json
```

### 不可逆操作：dry-run + 确认

回滚、删除等操作必须两步：先 dry-run 展示影响范围，用户确认后才执行。AI 替用户做不可逆决策是越权。

### 两级路由守卫

复杂 Skill 的内部路由分为**准入判断**和**路由选择**两层，禁止隐式回退——不同路径的数据来源和口径完全不同，自动切换会产生不一致数据且用户不知情。

## 六、Prompt 工程

### 角色设定匹配功能定位

- 诊断/分析型 Skill → 强人格角色（"你是一位资深优化师"），让 AI 主动推理
- 执行/工具型 Skill → 去人格化（"直接输出，不说铺垫"），抑制创造性发挥

错配角色会导致执行型 Skill 擅自添加不需要的分析，分析型 Skill 变成被动问答机。

### 隐性知识用决策矩阵

专家判断逻辑用表格而非自然语言。表格强制穷举多变量组合，把"需要经验的推理"降级为"查表"：

| 输入A | 输入B | 输入C | 诊断结论 |
|:-----:|:-----:|:-----:|---------|
| 稳定 | ↑ | ↓ | 结论1 |
| ↑ | ↑ | ↓ | 结论2 |

### 量化替代模糊

所有可量化的约束都必须量化。"合理控制"对 AI 可以是 1 次也可以是 10 次，"最多重试 1 次，仅限超时错误"才是可执行的规则。

### 对话推进靠发现

分析型 Skill 中，禁止 AI 用提问替代行动。先用合理默认值查数据，带着发现推进对话，而非让用户填表单。

### 优先级分层声明

多条规则可能冲突时，标注优先级层级。LLM 无法自动判断规则优先级，没有声明则随机遵守。

---

## 七、防滥用与凭证

### 频控双锁

消息发送类 Skill 同时设间隔限制和总量限制。单锁都有漏洞。

### 单一鉴权入口

所有脚本通过统一鉴权模块获取 token，禁止散落在各脚本中。安全审计一处检查，token 轮换一处生效。

### 敏感信息遮蔽

含 token 的 URL 用 markdown 锚文本输出，凭证展示时遮盖中间部分。AI 输出直接呈现在界面上，可能被截图分享。

---

## 审查检查清单

创建初版时逐项对照确保不遗漏，迭代时逐项验证查漏补缺：

**结构层**
- [ ] Front Matter 包含 name + description（含触发和不触发条件）
- [ ] 核心规则/禁止项放在文档前部，优先于 SOP
- [ ] SKILL.md 总行数在 150-400 行，超出部分拆到 references/
- [ ] references/ 中的文件在 SKILL.md 中有条件引用指引

**触发层**
- [ ] 正向触发用意图动词 + 原话举例
- [ ] 有负向排除条件，且标注了排除后"留给谁"
- [ ] 模糊场景有默认路由，不留给运行时猜测
- [ ] 与同项目下其他 Skill 无触发域冲突

**职责层**
- [ ] AI 与脚本的职责边界明确——AI 不做计算/渲染
- [ ] 脚本调用命令完整可执行，非伪代码
- [ ] 多步流水线有成功信号约定
- [ ] 产物文件命名可预测

**安全层**
- [ ] 禁止条款有三层结构（禁止 + 原因 + 替代）
- [ ] 已知的 AI 缺陷用历史案例写进 prompt
- [ ] 不可逆操作有 dry-run + 确认机制
- [ ] 无静默降级——失败时显式告知用户
- [ ] 敏感信息（token/URL/凭证）有遮蔽规则

**Prompt 层**
- [ ] 角色设定与 Skill 类型匹配（分析型 vs 执行型）
- [ ] 专家判断逻辑用决策矩阵而非自然语言
- [ ] 量化约束替代模糊描述
- [ ] 规则有优先级分层声明
